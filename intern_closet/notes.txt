MIGRATION PATTERN
-> voir img dans intern closet

Napi::Env porte le contexte. En V8, on devait souvent écrire isolate->GetCurrentContext() pour obtenir le contexte courant. En node-addon-api, Napi::Env contient déjà cette information, donc tout est simplifié.

RAII au lieu de Local<T> + HandleScope. Les types Napi gèrent leur propre durée de vie. On ne déclares plus de HandleScope scope(isolate) en début de fonction, le wrapper s'en occupe automatiquement via Napi::CallbackInfo.

Les valeurs portent leur env. Napi::Value, Napi::Object, etc. contiennent une référence implicite à leur Napi::Env. Donc dès qu'on passe une Napi::Value à une fonction, pas besoin de lui passer aussi l'env — elle est accessible via .Env().

Moins d'indirections. Plus de Template intermédiaire, plus de Context explicite — l'API expose directement le niveau où on travailles.

Les erreurs passent par exceptions, pas par types encapsulés. MaybeLocal<T> + .ToLocalChecked() + Maybe<bool> + .FromJust() → tout disparaît au profit du modèle exception standard C++. Ton code ressemble à du code C++ normal au lieu de ressembler à du Java verbalisé.

Les conversions courantes sont implicites. const char* → Napi::String, int → Napi::Number, etc. On as plus à envelopper chaque valeur primitive avant de la passer.

isolate->GetCurrentContext()->Global() → env.Global(). Accès au global en une étape.

obj->Get(context, FromUtf8(isolate, "key")).ToLocalChecked() → obj.Get("key"). Les accès de propriétés par clé string sont plus directs.

Les casts .As<T>() restent nécessaires quand on veux passer d'une Napi::Value générique à un type précis (Napi::Object, Napi::Function, etc.). Pas de magie.

function.Call(recv, std::vector<napi_value>) remplace le couple argc/argv. Les initializer lists {val1, val2} marchent directement.

Les erreurs se propagent par exceptions C++ au lieu de crasher le process via .ToLocalChecked(). C'est plus sûr par défaut.

v8::TryCatch + HasCaught() → try { ... } catch (const Napi::Error& e) { ... }. Le modèle V8 "exception en attente" disparaît au profit du modèle C++ standard.

tryCatch.Exception() → e.Value(). Récupère la valeur JS sous-jacente à l'exception.

IsNativeError() n'a pas d'équivalent direct. On teste .Get("stack").IsString() comme substitut pragmatique.

GetConstructorName() → .Get("name"). On passe par la propriété JS standard.

Exception::Error(msg) → Napi::Error::New(env, msg). Construction d'un objet Error JS depuis C++.

exc->Set(context, key, value).FromJust() → obj.Set(key, value). Même simplification que pour les autres accès de propriétés, plus de context explicite, plus de déballage.

isolate->ThrowException(error) → error.ThrowAsJavaScriptException(). L'idiome node-addon-api pour lever une exception JS depuis C++.

Le mode NAPI_CPP_EXCEPTIONS simplifie NatTryCatch mais ne le rend pas inutile. Les exceptions C++ non-Napi (SkipException, std::exception) doivent encore être attrapées et converties manuellement.

napiError.Value() retourne directement une Napi::Object parce qu'une Error JS est un objet. Pas besoin de cast .As<Napi::Object>().

pour fromjs, Persistant object :
// V8
static Persistent<Object> kExternFunctions;
kExternFunctions.Reset(isolate, externFunctions);  // pour assigner
Local<Object> obj = kExternFunctions.Get(isolate);  // pour récupérer

// Napi
static Napi::ObjectReference kExternFunctions;
kExternFunctions = Napi::Persistent(externFunctions);  // pour assigner
Napi::Object obj = kExternFunctions.Value();  // pour récupérer

aucun equivalent dans la nouvelle api a Isolate* isolate = Isolate::GetCurrent() , donc on va recuperer l'env depuis la reference elle meme, car
Napi::ObjectReference porte son propre env.

1 - Napi::Env remplace à la fois Isolate* et le Context. Quand on voyais isolate->GetCurrentContext()->machin(), maintenant juste env.machin().
et aussi les objets (style Napi::Value, Napi::Object, etc.) contiennent maintenant une référence implicite à leur Napi::Env.
-----------------

2 - Les erreurs passent par exceptions C++ standard. Plus de valeurs "peut-être absentes" à déballer, plus de TryCatch en attente — du vrai try/catch C++.

Anciennement en v8 pour les erreurs structurelles (propriete manquante, cast impossible), ca retournait MaybeLocal qui pouvait etre soit vide soit rempli, en fonction de si ca avait fonctionné en js ; il y avait des ToLocalChecked() partout mais sans verifier avant que c'etait rempli, bcp de crash potentiels.

Et pour les exceptions javascript explicites, vous utilisiez des try catch apres un call pour savoir si il avait echoué (enfin si il y a eu une exception) avec genre .hasCaught, donc ca oblige a faire des try catch systematiquement apres chaque call...

Maintenant, tout se simplifie beaucoup : toute exception coté JS devient une exception coté C++ grace au type Napi::Error, donc un seul try catch, try le call, catch Napi::Error.
et pour le cas MaybeLocal, il disparait, et a la place il renvoie soit Napi::Value soit il throw une Napi::Error.
-----------------

3 - Beaucoup de méthodes Napi acceptent directement les types C++ bruts (const char*, std::string, int, double). Là où V8 forçait à passer par un objet Local<String> ou Local<Number> intermédiaire, Napi propose des surcharges qui font la conversion en interne. Le code devient plus court et plus lisible.
-----------------

4 - Pour convertir une Napi::Value générique en type précis, deux outils : .As<Napi::Machin>() quand on est sûr du type (ou après un test IsMachin()), et .ToString() / .ToNumber() quand on veux forcer la conversion avec coercion.
-----------------

5 - Disparition des helpers de conversion manuels (FromUtf8...). Les constructeurs Napi (Napi::String::New, etc.) font le job directement.
-----------------

6 - Disparition des HandleScope manuels. En V8, chaque fonction qui manipulait des objets JS devait ouvrir un HandleScope en début de fonction pour structurer la durée de vie des références côté garbage collector. En Napi, ces scopes sont gérés automatiquement par le wrapper. Ca fait une ligne en moins par fonction sur tout le code, et un risque de bug en moins (oublier le handlescope)
-----------------

7 - Les exceptions C++ Napi reposent sur la macro NAPI_CPP_EXCEPTIONS (activée dans binding.gyp). Cette option permet aux erreurs JS d'être propagées comme exceptions C++ standard. Sans cette option, il faudrait revenir à un modèle proche de V8 (vérification manuelle de IsExceptionPending après chaque appel).
-----------------

8 - Napi exige un retour de Napi::value ; 
Dans v8, on utilisait args.GetReturnValue().Set pour return, en mettant le result dans un objet filé par le moteur, et donc jamais de return, toujours void
Maintenant, on peut return directement, et Napi attend un retour dans tout les cas, donc meme sur du void pur sans retour, ca devient un Napi::value, quitte a retourner env.Undefined()

En gros il y avait un probleme d'incoherence entre la declaration de la fonction coté cpp et sa vraie signature coté skip.
Côté C++; on déclarait SKIP_SKJSON_createCJInt(int64_t v) — comme si la fonction prenait un entier.
Mais en allant voir le code Skiplang dans Extern.sk, j'ai vu qu'elle est définie comme prenant un Float, pas un Int.

Vu que les int et les float ne passent pas par les memes registres, int64 passe dans RDI, double passe dans XMM0. Cpp envoyait la valeur dans RDI, skip allait la chercher dans XMM0 qui ducoup contenait n'importe quoi, et qui interpretait ca en 0. 

Pour confirmer, j'ai envoyé 0xDEADBEEF et j'ai vu qu'elle arrivait quand meme comme 0, donc les bits ne sont pas tronqués mais completement perdus en route, et ca explique aussi pourquoi
les strings passaient sans probleme, parceque un char* est un pointeur, donc passé dans le meme registre entier des deux cotés.
Pour fix, j'ai passé int64 en double dans la declaration extern C, et dans CreateCJInt on converti en double au lieu de int64.

ca impacte quoi/qui


Si je comprends bien, aujourd'hui une Resource peut utiliser les collections de createGraph et les services externes, mais pas d'autres Resources du même service.

Il faudrait qu'une resource puisse utiliser/dependre d'autres resources du meme service ?

Voila comment on l'utiliserait dans l'idée

class TopUsers implements Resource<Collections> {
  static uses = {
    stats: UserStats,
  };

  instantiate(cs, context) {
    const stats = context.useResource(TopUsers.uses.stats);
    return stats.map(...);
  }
}

Cette ressource a le droit d’utiliser UserStats.

TopUsers dépend de UserStats.

Si UserStats est modifiée, TopUsers est potentiellement impactée.

Si TopUsers est instanciée, UserStats doit être instanciable aussi, même sans client direct.


En gros si j'ai bien compris la demande, il faut que je rajoute la possibilité de dependances internes entre ressources d'un meme service.
Aujourd'hui, une ressource expose une collection calculée a partir de createGraph. L'idee serait qu'une ressource puisse declarer d'autres ressources du
meme service dans genre un "static uses", puis les recuperer via context.useResource, un peu comme useExternalRessource, mais en interne.


Impacts ?

- Quand B utilise A et que A change, B doit logiquement etre recalculé
- Si personne ne consulte A directement mais que B l'utilise, A va devoir etre recalculé aussi
- Il va falloir faire gaffe aux boucles infinies de dependence genre A qui veut utiliser B qui veut utiliser A...

Aujourd'hui, Context sait utiliser une ressource externe avec useExternalResource, mais pas une ressource du même service. Dans ContextImpl, on voit bien useExternalResource(...) et mon useLazyExternalResource(...), mais pas de useResource(...) interne.

On va toucher a :

- L'API publique, il va falloir modifier le Context pour ajouter useResource du genre
useResource<K extends Json, V extends Json>(
  resource: ResourceConstructor,
  params?: Json
): EagerCollection<K, V>;

- il va falloir ajuster le type des constructeurs de ressources pour supporter une proprieté statique uses

- Runtime interne, il va falloir une version interne de useExternalResource, mais pour le meme service

- La propagation des changements est gerée par SKStore
Si UserStats change, il ne suffit peut-être pas de recharger UserStats : il faut aussi recharger toutes les ressources qui dépendent d’elle, par exemple TopUsers.

- Gestion du cycle de vie ?
Aujourd'hui une ressource est gardée vivante tant qu'un client externe est abonné, sinon elle part au garbage. Une ressource interne n'a pas de client externe, donc il faut un nouveau critère pour décider quand la supprimer.
Donc UserStats peut être “non consultée directement”, mais doit rester vivante tant que TopUsers en dépend.

- Trois nouveaux types :
ResourceDependency qui décrit une dépendance unitaire (le nom de la ressource cible et la fonction de transformation des params)

ResolvedDependencies qui est le pendant côté instanciation : ce que la classe reçoit en argument une fois que le runtime a fait le boulot de résolution.

- Nouveau champ DependentResources dans SkipService, champ optionnel
DependentResources est comme resources, une map ou chaque cle string pointe vers un constructeur de dependentResource, mais la classe doit aussi avoir une propriete statique dependencies (sinon ts ne nous laissera pas y acceder, on serait obligé de foutre des as any).
DependentResource est une interface séparée et pas un héritage de Resource, parce que sa méthode instantiate a 3 arguments (collections, dependencies, context) au lieu de 2.

index.ts
- Ajout d'un algo de parcours en profondeur, DFS, systeme de detection de cycles style flood-fill mais avec 3 etats visité et valide/ pas visité / en cours de visite, appelé au constructeur de SkipService,
qui verifie si les dependentResources ont des dependences problematiques qui forment des cycles. On verifie que c'est un DAG
DAG = Directed Acyclic Graph

- Le DFS en profite aussi a chaque noeud pour verifier que la dependence cible existe quelque part, pour eviter qu'une dependentResource ne declare une dependence a une resource 
qui n'existe nulle part, trois cas :
La cible est dans dependentResources → c'est une DependentResource, on continue le parcours récursif sur elle.
La cible est dans resources (et seulement là) → c'est une Resource simple, c'est une feuille du graphe, on s'arrête là sans erreur.
La cible n'est nulle part → erreur claire au boot.

A FAIRE

Il faut encore que je gére l'instanciation des DependentResource. Aujourd'hui, le binding SkipRuntime_Resource__instantiate côté ToBinding reçoit un handle vers une instance de Resource et appelle son instantiate. Cette mécanique ne fait pas la distinction entre une instance de Resource et de DependentResource, et même si elle le pouvait, elle ne dispose ni du nom de la ressource ni de ses paramètres parents pour orchestrer la résolution des dépendances.

Je compte ajouter un binding parallèle dédié SkipRuntime_DependentResource__instantiate côté index.ts, avec sa déclaration dans binding.ts, son export Skiplang dans FFI.sk, et l'aiguillage côté Runtime.sk dans buildResourcesGraph pour reconnaître le type de ressource et appeler le bon binding.

Pour la résolution des dépendances elle-même, il faut que les sous-ressources soient inscrites dans la chaîne de dirs SKStore (kResourceSessionDir → kResourceAvailablesDir → kResourceCollectionsDir) pour bénéficier du mécanisme de déduplication et de garbage existant.  Le souci, c'est que kResourceSessionDir n'autorise qu'une seule ressource par identifier — donc si on essaie de réutiliser l'id du parent pour la sous-ressource, ça collisionne avec l'inscription du parent lui-même.

J'ai remarqué que kResourceInstance2SubIdDir est défini dans Utils.sk mais n'est utilisé nulle part dans les fichiers que j'ai. Le nom suggère un mapping parentInstanceId → subIds, ce qui ressemble très exactement au mécanisme dont on a parlé pour gérer les sous-ressources internes sans collision avec les sessions clients.

MODIF INSTANCIATERESOURCE

Avant d'appeler le binding pour créer la ressource, on appelle une fonction preloadDependencies.
preloadDependencies lit la classe via getResourceClass (méthode que j'ai ajouté a ServiceDefinition)
Si la classe n'existe pas ou n'a pas de static dependencies, on retourne immédiatement.
Pour chaque dépendance, on calcule les params via la fonction de transformation et on construit un id par suffixage (${parentId}__${depName}).
On appelle récursivement instantiateResource sur la dépendance. Si la dépendance est elle-même une DependentResource, sa propre préphase se déclenchera, et ainsi de suite.
À la fin de preloadDependencies, toutes les sous-ressources sont inscrites et calculées.
Seulement après, on appelle le binding SkipRuntime_Runtime__createResource pour la ressource finale.
